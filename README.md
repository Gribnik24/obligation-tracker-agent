# Obligation Tracker Agent

AI-агент на базе LangGraph для анализа регулярных платежей, подписок и финансовых обязательств. Агент умеет фильтровать данные по категориям и датам, а также конвертировать валюты в единый формат (RUB) для корректного подсчёта итогов, а также имеет память контекста диалога.

## Структура проекта

- `main.py` — Точка входа, цикл взаимодействия с пользователем.
- `src/agent.py` — Граф LangGraph, узлы агента и роутинг.
- `src/tools.py` — Реализация инструментов (время, обязательства, валюта).
- `config/` — Конфигурация (настройки, системный промпт).
- `data/` — База данных подписок (JSON).
- `tests/` — Unit-тесты для инструментов.

## Как запустить агента

Проект полностью контейнеризован. Для работы требуется установленный **Docker** и **Docker Compose**.

### 1. Настройка окружения

Создайте файл `.env` (Пример присутствует в файле `.env.example`) в корневой директории проекта и укажите параметры для LLM:

```env
# Настройки LLM (через OpenRouter)
MODEL_NAME = tencent/hy3:free
MODEL_API_BASE = https://openrouter.ai/api/v1
MODEL_API_KEY = sk-or-v1-xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> **Примечание 1:** Вы можете использовать любую модель, доступную через OpenRouter, изменив значение `MODEL_NAME`.

> **Примечание 2:** Запуск системы из России следует вопроизводить с включенным VPN.

### 2. Запуск агента

Соберите образ и запустите интерактивный режим:

```bash
docker-compose run --rm agent
```

После запуска в терминале появится приглашение `USER:`. Вводите вопросы на русском языке, например:
- "Сколько я потрачу в ближайшие 30 дней?"
- "Какие у меня платежи на этой неделе?"
- "Сколько я трачу на подписки?"

Для выхода из агента введите `/exit`.
Для сброса контекста диалога введите `/reset`.

## Выбор платформы LLM

**Используемая платформа:** [OpenRouter](https://openrouter.ai/).

**Преимущества решения:**
1. **Единый API:** Позволяет использовать модели разных провайдеров (OpenAI, Google, Meta и др.) через один интерфейс.
2. **Гибкость:** Можно легко переключаться между моделями, тестировать их, меняя одну переменную в `.env`, без изменения кода агента.

## Выбор LLM

### Tencent: Hy3 (free)

**Используемая модель:** [Tencent: Hy3 (free)](https://openrouter.ai/tencent/hy3:free) (в файле `.env` `MODEL_NAME=tencent/hy3:free`)

**Преимущества решения:**
1. Модель имеет 295 миллиардов параметров, но благодаря архитектуре MoE активирует только 21 миллиард из них для обработки каждого запроса. Это обеспечивает высокую производительность при относительно низких вычислительных затратах, что критично при пакетной обработке больших массивов финансовых данных.

2. Поддерживает настраиваемый режим "размышлений" (chain-of-thought). Для финансовых задач это позволяет модели поэтапно распутывать сложные многофакторные взаимосвязи (например, влияние макроэкономических показателей на стоимость актива) и делать выводы с прозрачной логикой.

3. Поддерживает контекст до 256K токенов.

4. Модель полностью бесплатна.

5. Разработчики делают акцент на надежности и борьбе с галлюцинациями. В финансах это критично: модель будет отмечать недостаток данных для ответа, а не генерировать несуществующие цифры, что снижает риски принятия ошибочных инвестиционных решений.

6. Стабильная работа с вызовом внешних инструментов (Tool Calling). Это позволяет интегрировать модель в агентные системы для автоматического сбора данных с финансовых API (например, получение текущих курсов или балансов) и последующего анализа.

> Эта модель будет недоступна с 21.07.2026. Вероятно, разработчик выложил ее в тестовом бесплатном режиме для проверки работоспособности. Поэтому дополнительно для тестирования была выбрана еще одна модель

[Примеры полных ReAct-трейсов модели Tencent: Hy3 (free)](react_trace_Hy3.md) (дата формирования: 08.07.2026)

## Ограничения

Агент разработан как демонстрационный прототип и имеет следующие ограничения:

1. **Статичные данные**
   - Агент не подключается к реальным банковским API. Данные о платежах берутся из локального файла `data/subscriptions.json`.
   - Для добавления новых платежей необходимо вручную редактировать JSON-файл.

2. **Отсутствие исполнения платежей**
   - Агент умеет только *анализировать* и *считать* расходы - его задача как раз только в этом. Он не может оплачивать счета или создавать новые транзакции в банке.

3. **Зависимость от внешних API**
   - Конвертация валют зависит от API `frankfurter.dev`. При недоступности этого сервиса агент не сможет рассчитать итоговую сумму в рублях.

4. **Память**
   - Контекст диалога хранится в памяти процесса. При перезапуске контейнера история переписки теряется.

5. **Фильтрация по деньгам**
   - У агента отсутствует прямая функция фильтрации платежей по сумме (меньше или выше какой-то конкретной сумме) по причине того, что в зависимости от типа платежа в поле `currency` будут указаны разные валюты. Выполнение подобной фильтрации потребует перевод каждой суммы в фиксированную валюту и дальнейшее сравнение.

6. **Формат данных. Учет частоты выплат**
   - Один из тестовых запросов звучит как «Какая категория обходится мне дороже всего?» Без специального поля в данных платежей, который бы отвечла за частоту выплат (пр. `annual`, `monthly`), агент не сможет на основе конкретно предложенных данных без своих допущений выдать точный ответ. Например, какая-то выплата довольно дорогая, но приходится всего один раз в год, в то время как другая подписка имеет более дешевую месячную стоимость, которая при подсчете за год выдаст бОльшую сумму. Агент же признает самой дорогой подпиской ту - где присутствует один (как оказывается, годовой) большой платеж.

7. **Формат данных. Актуальные категории**
   - Актуальные категории выплат (поле `category` в `data/subscriptions.json`) напрямую прописываются в одном из инструментов агента (описание `get_obligations` и системный промт). Если в данных появится новая категория, то ее будет необходимо также добавлять в вышеуказанные элемент агентной системы.
